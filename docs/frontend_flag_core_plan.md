# Frontend Flag Core Modularization Plan

## Summary

This plan starts frontend modularization with the area that changes most often: llama.cpp flag definitions, flag state, launch argument generation, and Configure tab flag rendering.

The goal is to make flag additions, removals, renames, default changes, and option updates easier to maintain while preserving the UI state sync rule from `AGENTS.md`.

Primary rule: Configure, Quick Launch, Chat sampler controls, presets, and command preview must continue to read and write through one shared flag state path.

## Phase 0: Baseline And Guardrails

Status: In progress

- [x] Record current implementation baseline before extraction:
  - `ui/js/flags.js` defines all supported flag input types: bool, int, float, text, path, enum, and multi_enum.
  - `ui/js/app.js` currently owns shared state and launch behavior through `currentTool`, `flagValues`, `setFlagValue()`, `setPathFlagValue()`, `setMultipleFlagValues()`, `collectFlagValues()`, `applyFlagValues()`, `updateCommandPreview()`, and `getLaunchArgs()`.
  - `syncUiAfterSharedStateChange()` currently refreshes Configure inputs, command preview, and Chat sampler UI after shared state changes.
  - Configure rendering currently lives in `renderFlags()`, `createSubmenuBlock()`, `createFlagRow()`, and `restoreFlagInputs()`.
  - Quick Launch writes model, context, GPU, template, sampler, and metrics controls through the shared setter path.
  - Chat sampler sliders write through `setFlagValue()`.
  - Preset load applies state through `applyFlagValues()`, and preset save/export reads through `collectFlagValues()` via the preset data helpers.
  - Command preview is generated by `getLaunchArgs()` from shared `flagValues`.
- [x] Add or document a manual smoke checklist for the workflows above.
- [x] Keep the first implementation behavior-neutral:
  - no framework migration
  - no `type="module"` conversion
  - no UI redesign
  - no intentional launch argument changes
- [ ] Run manual browser smoke checklist before Phase 1 extraction. Started; remaining unchecked items still need a conclusive pass.

## Phase 1: Create A Flag State And Launch Core Boundary

Status: Not started

Create a dedicated shared runtime boundary, initially compatible with classic script loading and global access.

Recommended file: `ui/js/flag-core.js`

The flag core should own:

- [ ] `currentTool`
- [ ] `flagValues`
- [ ] `setCurrentTool`
- [ ] `setFlagValue`
- [ ] `setPathFlagValue`
- [ ] `setMultipleFlagValues`
- [ ] `collectFlagValues`
- [ ] `applyFlagValues`
- [ ] `shouldOmitFlagValue`
- [ ] `getLaunchArgs`
- [ ] `updateCommandPreview`

Implementation constraints:

- [ ] Expose the API through `window.LlamaGui.flagCore`.
- [ ] Keep compatibility shims for old global function names until all callers are migrated.
- [ ] Keep all launch args generated from `flagValues`, not DOM-local copies.
- [ ] Keep `setMultipleFlagValues()` as the single write path for grouped flag updates.
- [ ] Keep one post-update refresh hook that updates mirrored UI and command preview.

Expected result:

- `app.js` still orchestrates the app, but shared flag state and launch argument generation have one obvious home.

## Phase 2: Add Flag Definition Validation

Status: Not started

Add lightweight validation around `ui/js/flags.js` so common llama.cpp flag update mistakes are caught early.

Recommended file: `ui/js/flag-validation.js`

Validation should check:

- [ ] every flag has a unique `id`
- [ ] every flag has a valid `category`
- [ ] every flag has a valid `tool`: `server`, `cli`, or `both`
- [ ] every flag has a supported `type`
- [ ] `enum` and `multi_enum` flags have usable `options`
- [ ] `false_flag` only appears on `bool` flags
- [ ] default values are compatible with the declared type where practical
- [ ] duplicate CLI flag strings are reported unless intentionally allowed

Implementation constraints:

- [ ] Run validation once during frontend startup.
- [ ] Report validation problems to the browser console.
- [ ] Do not block app startup in the first version.
- [ ] Avoid network access or upstream llama.cpp checks in this phase.

Expected result:

- Updating `flags.js` becomes safer without changing runtime behavior.

## Phase 3: Extract Configure Flag Rendering

Status: Not started

Move Configure tab flag rendering into a focused module after the shared flag core is stable.

Recommended file: `ui/js/config-flags-ui.js`

This module should own:

- [ ] `renderFlags`
- [ ] `createSubmenuBlock`
- [ ] `createFlagRow`
- [ ] Configure search filtering
- [ ] expand/collapse behavior
- [ ] `restoreFlagInputs`
- [ ] `normalizeMultiEnumValue`
- [ ] high-risk multi_enum warning display

Split `createFlagRow()` into type-specific builders:

- [ ] bool row builder
- [ ] enum row builder
- [ ] multi_enum row builder
- [ ] path row builder
- [ ] int row builder
- [ ] float row builder
- [ ] text row builder

Implementation constraints:

- [ ] All controls must read current values from `window.LlamaGui.flagCore`.
- [ ] All controls must write through the shared setters.
- [ ] No Configure control may keep a separate copy of launch-relevant state.
- [ ] Search and expand/collapse state may remain local to the Configure UI module.

Expected result:

- Adding or tweaking a flag definition should usually require changing only `flags.js`.
- Adding support for a new flag input type should have one clear rendering location.

## Phase 4: Migrate Callers And Remove Compatibility Shims

Status: Not started

Once the flag core and Configure renderer are stable, migrate direct references in the remaining frontend code.

Areas to update:

- [ ] Quick Launch controls
- [ ] Chat sampler controls
- [ ] sampler preset apply/load/save behavior
- [ ] preset import/export behavior
- [ ] launch/stop flow
- [ ] API URL/model helpers
- [ ] stats and metrics helpers that read host/port

Implementation constraints:

- [ ] Migrate one area at a time.
- [ ] Run the smoke checklist after each area.
- [ ] Remove old global compatibility shims only after no callers remain.

Expected result:

- The shared flag core becomes the single supported API for flag reads, writes, and launch args.

## Acceptance Criteria

- [ ] `flags.js` remains the single source of truth for exposed llama.cpp flags.
- [ ] `flagValues` has one owner.
- [ ] Every duplicated control updates through the shared setter path.
- [ ] Command preview and launch arguments are generated from shared state only.
- [ ] Configure rendering is isolated from Quick Launch and Chat behavior.
- [ ] New or renamed llama.cpp flags can be added with minimal changes outside `flags.js`.
- [ ] Existing user-facing behavior remains unchanged unless intentionally documented.
